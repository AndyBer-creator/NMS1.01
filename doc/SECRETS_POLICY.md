# NMS1 Secrets & Env Policy

Политика хранения и ротации секретов для production.

## 1) Что считается секретом

- `POSTGRES_PASSWORD`, `DB_DSN` (если содержит пароль)
- `NMS_ADMIN_PASS`, `NMS_VIEWER_PASS`
- `NMS_SESSION_SECRET`
- `TELEGRAM_TOKEN`, `TELEGRAM_CHAT_ID`
- `SMTP_PASS` (+ при необходимости `SMTP_USER`)

## 2) Базовые правила

1. `.env` не коммитится в git (только `.env.example`).
2. Секреты хранятся только в защищённом storage хоста/CI (или Docker secrets/Vault).
3. В логах и ошибках нельзя печатать значения секретов.
4. Любое подозрение на утечку = немедленная ротация.

## 3) Ротация (минимум)

- `NMS_SESSION_SECRET`: каждые 90 дней
- `NMS_ADMIN_PASS`, `NMS_VIEWER_PASS`: каждые 60–90 дней
- `SMTP_PASS`: по политике почтового провайдера, но не реже 90 дней
- `TELEGRAM_TOKEN`: при утечке/инциденте немедленно, планово раз в 180 дней
- `POSTGRES_PASSWORD`: не реже 180 дней (или чаще по требованиям)

## 4) Процедура ротации

1. Подготовить новые значения.
2. Обновить env на прод-хосте.
3. Перезапустить затронутые сервисы:
   ```bash
   docker compose -f deploy/compose/docker-compose.yml up -d --build api worker trap-receiver
   ```
4. Прогнать проверки:
   ```bash
   make smoke-test
   make rbac-smoke
   ```
5. Проверить alert pipeline (email/telegram).
6. Зафиксировать дату ротации в журнале ops.

## 5) Инцидент утечки

Если секрет утёк:
1. Отозвать/сменить скомпрометированный ключ или пароль.
2. Перезапустить сервисы с новыми секретами.
3. Проверить доступность системы и авторизацию.
4. Оформить postmortem с причиной и мерами.

## 6) Путь к целевому состоянию

Текущий реализованный шаг: Docker-secrets процесс с чтением секретов через `*_FILE` и операционной процедурой (`SECRETS_PROCESS.md`).

Следующий уровень зрелости:
- перейти на централизованное secret-хранилище (Vault / cloud secret manager),
- внедрить автоматическую ротацию и аудит доступа к секретам.
